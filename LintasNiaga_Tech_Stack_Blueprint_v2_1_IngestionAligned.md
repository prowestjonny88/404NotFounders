# LintasNiaga Tech Stack Blueprint
## Comprehensive Technical Stack & Implementation Guide

**Project:** LintasNiaga  
**Document Type:** Hackathon implementation blueprint  
**Version:** 2.1 (Updated for ingestion-first backend and latest data-source changes)  
**Purpose:** Translate the latest PRD and architecture into a stable, buildable, hackathon-ready technical stack.

---

## 1. Technical Strategy Summary

LintasNiaga should be built as a **single monorepo** containing **two primary application services** and one clearly defined **data/snapshot layer**.

1. **Web application** for user interaction, upload flow, comparison UI, charts, and analyst output
2. **API service** for deterministic business logic, ingestion orchestration, validation, simulation, ranking, and persistence
3. **Snapshot/data layer** for reference JSON, raw artifacts, and normalized market/logistics snapshots generated by the backend

### Core Architecture Principle

> **Deterministic backend first, ingestion layer second, agentic wrapper third.**

This means:
- the browser is thin,
- FastAPI is the application backbone,
- ingestion pipelines normalize external context before analysis uses it,
- AI orchestration is thin and bounded,
- and all business-critical rules stay in deterministic backend services.

### What the stack should optimize for

- demo stability
- speed of implementation
- clear ownership between frontend, backend, and ingestion
- easy debugging
- explicit schema boundaries
- graceful degradation
- judge-friendly observability
- low fragility from external data sources

## 2. Final Stack Decisions

## 2.1 Frontend

- **Framework:** Next.js (App Router)
- **Language:** TypeScript
- **Styling:** Tailwind CSS
- **UI Components:** shadcn/ui
- **Charts:** Recharts by default
- **Custom chart escape hatch:** D3 only if one chart later truly requires bespoke interaction
- **Forms:** React Hook Form + Zod
- **Client state:** local React state only where needed
- **Server state:** TanStack Query
- **Upload:** multipart file upload from frontend to FastAPI
- **Streaming UI:** SSE consumption for analyst/status output only

### Why this still holds
The new ingestion-first backend does not change the frontend philosophy.  
The UI should still remain thin and render:
- upload and repair flows,
- quote comparison,
- recommendation cards,
- analyst output,
- hedge what-if UI,
- and status for ingestion freshness if needed.

---

## 2.2 Backend

- **Language:** Python 3.11
- **Framework:** FastAPI
- **Validation / schema layer:** Pydantic v2
- **HTTP client:** httpx
- **Retry / resilience:** tenacity
- **Serialization:** orjson
- **Async file support:** aiofiles
- **Tabular/time-series handling:** pandas + numpy
- **AI orchestration:** LangGraph (thin orchestration only)
- **AI provider integration:** one model-provider wrapper module
- **Market data ingestion:** provider adapters + snapshot pipelines
- **Static/reference data access:** centralized repository/data-access layer
- **Raw artifact handling:** local disk in hackathon mode

### Why this changed
The old stack assumed a simpler static-data + BNM model.  
The updated design introduces:
- provider adapters,
- ingestion jobs,
- snapshot normalization,
- raw artifact storage,
- and selective scraping.

That requires a slightly richer backend foundation, but still within a hackathon-safe envelope.

## 2.3 Ingestion and Data Source Stack

### 2.3.1 Local / Static Anchors

### 2.3.5 News / Event Context
- **GNews** Python library
- Use it to ingest structured headlines and convert them into event context snapshots

### 2.3.6 PP Resin Benchmark Extraction
- **Trafilatura** for text extraction from curated web pages
- **LLM extraction layer** for pulling structured benchmark price fields from extracted text
- **Deterministic validators** before promoting snapshots

## 2.4 Persistence

### Local / Hackathon Mode

- **Normalized snapshots:** local JSON files under `/data/snapshots`
- **Raw source artifacts:** local files under `/data/raw`

### Why this evolved
The earlier stack only distinguished static JSON vs SQLite.  
The updated system needs a third important layer:

- **reference anchors**
- **generated snapshots**
- **dynamic application state**

That separation is now central to the build.

## 3. Repo Strategy

## 3.1 Monorepo Shape

Use **one monorepo** with two clear apps and one explicit data area.

```text
lintasniaga/
  apps/
    web/              # Next.js frontend
    api/              # FastAPI backend
  data/
    reference/        # deterministic local anchors
    snapshots/        # normalized generated datasets
    raw/              # raw artifacts from ingestion/scraping
    tmp/
  packages/
    docs/             # optional shared docs/contracts
  scripts/
  .github/
  Makefile
  README.md
```

## 3.2 Why One Repo
A monorepo is still the best fit because:
- frontend and backend evolve together,
- ingestion contracts stay visible,
- schema changes stay coordinated,
- easier local startup and judge demo prep,
- fewer version-sync issues between repos.

## 3.3 Why Not Separate Repos
Separate repos increase coordination overhead exactly when architecture, ingestion, and contracts are still changing.

## 3.4 Why Not One Single Next.js App Only
The system now has too much backend-owned responsibility to hide inside route handlers or server actions:

- ingestion jobs
- external provider adapters
- snapshot normalization
- PDF extraction + validation
- landed-cost computation
- ranking logic
- hedge recalculation
- bounded AI orchestration

A proper FastAPI backend remains the correct boundary.

## 6. AI Architecture

## 6.1 Orchestration Philosophy

### Chosen
- thin LangGraph orchestration layer around deterministic FastAPI services

### Why
LintasNiaga is not primarily an autonomous agent system. It is a deterministic procurement decision engine with AI-enhanced steps.

So:
- FastAPI remains the backbone
- ingestion pipelines normalize context first
- LangGraph wraps only the parts where orchestration genuinely helps

---

## 6.2 What LangGraph Should Own

Only these AI-heavy steps:
- quote extraction flow coordination
- context assembly for reasoning
- reasoning / explanation flow
- streaming analyst output
- bounded recommendation-adjustment step

### Optional later
- controlled orchestration around the resin benchmark extraction pipeline, if it truly helps

---

## 6.3 What LangGraph Should NOT Own

Do not move these into the graph unless truly necessary:
- hard validation rules
- static data loading
- snapshot repository access
- market-data provider logic
- Monte Carlo simulation
- core ranking engine
- hedge calculation
- persistence

---

## 6.4 Model Provider Integration Rule

### Chosen
- one thin provider wrapper module for the model layer

The rest of the app should never talk to the provider SDK/client directly.

Example provider interface:
- `extract_quote_fields()`
- `reason_about_recommendation()`
- `stream_analyst_output()`
- `extract_resin_benchmark_from_text()`

This keeps provider-specific logic isolated.

### Reference JSON = deterministic anchors

### Snapshot JSON = generated context
Examples:
- FX histories

Examples:
- uploaded quote metadata

All data access should go through one repository/data-access boundary.

### Why
This prevents:
- duplicate file-read logic
- inconsistent fallback behavior
- silent schema drift
- direct provider leakage into services
- future migration pain when moving from local snapshots to DB/object storage

### Rule
Services should never read files directly.

## 20. What Is Explicitly Not Allowed in v1

To prevent stack drift, avoid all of the following unless absolutely necessary.

### Do not add
- microservices
- GraphQL
- Redux / RTK by default
- Zustand by default
- WebSockets by default
- frontend-owned business logic
- direct frontend-to-Supabase critical flows
- multiple model provider wrappers scattered through the app
- direct file reads inside services
- Celery / Redis / queue stack
- Docker-first complexity if local dev already works
- Turborepo as core repo infrastructure at hackathon stage
- enterprise observability stack
- unbounded generic web scraping
- live article scraping inside comparison requests

### Why
None of these are the bottleneck right now.  
The bottleneck is building a trustworthy recommendation pipeline with a stable ingestion layer.

## 21. Canonical Runtime Flow

This is the intended technical flow of a happy-path comparison.

1. **Frontend** uploads PDFs to FastAPI
2. **FastAPI** stores PDFs locally
3. **AI orchestration layer** invokes quote extraction
4. **Backend services** validate extracted quote data
5. invalid quotes are flagged for repair
6. user repairs invalid quote data through frontend form
7. repaired data is revalidated in FastAPI
8. comparison request is submitted
9. **backend services** load latest valid snapshots + reference anchors
10. **deterministic services** compute costs, penalties, and probabilistic outputs
11. **ranking engine** generates ranked quotes
12. **AI orchestration** assembles reasoning context and explanation
13. **bounded override step** may adjust winner only between top 2 if rules allow
14. **recommendation payload** is returned to frontend
15. **AI Analyst stream** optionally emits progressive analysis text
16. frontend renders:
    - table
    - recommendation card
    - backup option
    - analyst panel
    - hedge what-if UI

## 22. Canonical Ingestion Flow

This is the intended technical flow of a normal ingest refresh.

1. **route or job trigger** starts an ingest run
2. provider adapter fetches source data
3. service normalizes source payload into a typed schema
4. repository writes raw artifact if needed
5. repository writes normalized snapshot
6. manifest/status record is updated
7. later analysis runs consume the latest valid snapshot

Special case for resin benchmark:
1. fetch curated source HTML
2. store raw HTML
3. run Trafilatura extraction
4. store extracted text
5. LLM extracts structured price fields
6. validator checks schema + sanity rules
7. only then promote to latest valid benchmark snapshot

## 23. Final Technical Rule Set

These rules should be treated as non-negotiable for implementation.

1. One monorepo, two apps, one clear data area.
2. Frontend is thin; backend owns truth.
3. FastAPI is the application backbone.
4. LangGraph must stay thin.
5. REST by default; SSE only where it clearly helps.
6. Reference JSON, snapshot JSON, and SQLite state must remain distinct.
7. Local disk now, Supabase Storage later.
8. Supabase later is database + storage only; FastAPI stays primary backend.
9. One model provider wrapper only.
10. One provider adapter per external source type.
11. One centralized repository/data-access layer only.
12. Pydantic is backend schema truth.
13. Frontend uses TypeScript + Zod for critical forms/parsing.
14. React Hook Form powers repair/input forms.
15. TanStack Query is for server state only where useful.
16. Langfuse is only for AI traces; normal app logs handle everything else.
17. Ingestion must generate normalized snapshots before analysis consumes external context.
18. Scraping must happen in ingest/refresh flows, not live comparison requests.
19. Avoid stack bloat unless a concrete bottleneck appears.

## 24. Nuances and Uncertainties

These do not block the stack, but should remain visible.

1. **FX source is optimized for speed, not final institutional rigor**  
   `yfinance` is acceptable for hackathon speed, but should remain adapter-isolated.

2. **OpenWeatherMap adds real API-key friction**  
   Weather risk is useful, but the team should only model it as derived shipping risk.

3. **OpenDOSM dataset handling may vary by dataset shape**  
   Prefer simple CSV/Parquet ingestion and normalization over overengineering.

4. **GNews should remain metadata-first**  
   Headlines are event signals, not deterministic truth.

5. **Trafilatura is the highest-fragility component**  
   The hard part is source selection, validation, and promotion logic, not just extraction code.

6. **Snapshot freshness UX is optional**  
   The backend should support it, but the frontend may not need to expose it prominently in MVP.
